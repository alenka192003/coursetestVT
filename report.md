# Отчёт по курсовому проекту

## ВВЕДЕНИЕ

В рамках курсового проекта был разработан и проверен CI/CD-конвейер для HTTP-приложения, написанного на языке Go. Основная цель работы заключалась в том, чтобы реализовать приложение, разместить его код в git-репозитории, настроить автоматическую сборку при появлении git-тега, подготовить доставку приложения в Kubernetes и проверить доступность приложения по HTTP-запросам.

В качестве приложения был реализован небольшой HTTP API-сервис `coursevt`, предоставляющий несколько GET endpoint'ов. Приложение возвращает информацию о сервисе, статус работоспособности, список учебных курсов и данные конкретного курса по идентификатору. Это позволяет продемонстрировать не только базовую обработку HTTP GET-запросов, но и более структурированную серверную логику.

Для сборки приложения был подготовлен `Dockerfile`, позволяющий собирать контейнерный образ Go-приложения. Для развертывания в Kubernetes были созданы манифесты `Deployment`, `Service` и `Ingress`. Также была подготовлена отдельная локальная конфигурация Kubernetes для демонстрации работы приложения в локальном кластере Docker Desktop Kubernetes.

CI/CD-конвейер был реализован с использованием GitHub Actions. Workflow запускается при публикации git-тега формата `v*`, выполняет тестирование приложения, сборку Docker-образа, публикацию образа и последующий деплой в Kubernetes. Для доставки приложения в локальный Docker Desktop Kubernetes был настроен self-hosted GitHub runner, который запускается на локальной машине и имеет доступ к `kubectl`, Docker и Kubernetes context `docker-desktop`.

В ходе работы была проверена работоспособность Kubernetes-кластера, успешное создание pod'ов приложения, обработка HTTP GET-запросов через port-forward, а также доступность приложения по собственному локальному доменному имени `coursevt.local`.

## ХОД РАБОТЫ

Работа над проектом началась с создания структуры Go-приложения. Для проекта был создан модуль `coursevt`, после чего код был разделён на несколько логических частей. Такой подход был выбран для того, чтобы приложение не состояло из одного файла `main.go`, а имело более понятную архитектуру, близкую к реальной структуре backend-сервисов.

Основная точка входа приложения находится в файле `cmd/coursevt/main.go`. В этом файле выполняется только запуск приложения: загружается конфигурация, создаётся хранилище данных, инициализируется HTTP-сервер и запускается прослушивание порта. Вынесение точки входа в каталог `cmd` позволяет отделить исполняемый файл от внутренней логики приложения.

Конфигурация приложения была вынесена в пакет `internal/config`. В нём реализована загрузка параметров из переменных окружения. В проекте используются переменные `PORT` и `APP_VERSION`. Переменная `PORT` определяет порт, на котором запускается HTTP-сервер, а `APP_VERSION` используется для отображения версии приложения в ответах API. Если переменные окружения не заданы, используются значения по умолчанию: порт `8080` и версия `dev`.

Для описания предметной области был создан пакет `internal/course`. В нём определена структура курса, содержащая идентификатор, название, описание, продолжительность в часах и уровень сложности. Также был реализован интерфейс хранилища и in-memory реализация, которая содержит несколько заранее подготовленных учебных курсов. Такой вариант подходит для демонстрационного проекта, так как не требует подключения базы данных, но при этом позволяет показать работу с данными и отдельный слой приложения.

HTTP-слой был вынесен в пакет `internal/httpserver`. В нём реализованы маршруты, обработчики запросов, формирование JSON-ответов и обработка ошибок. Для маршрутизации используется стандартный пакет Go `net/http`, что позволило не добавлять внешние зависимости. Все ответы приложения возвращаются в формате JSON, а для неподдерживаемых HTTP-методов возвращается статус `405 Method Not Allowed`. Для несуществующих маршрутов и отсутствующих курсов возвращается статус `404 Not Found`.

В результате было реализовано HTTP API-приложение `coursevt`, которое обрабатывает GET-запросы и возвращает ответы в формате JSON. В проекте были реализованы следующие endpoint'ы:

- `GET /` - получение общей информации о сервисе, версии приложения и доступных маршрутах.
- `GET /healthz` - проверка работоспособности приложения.
- `GET /api/v1/courses` - получение списка учебных курсов.
- `GET /api/v1/courses/{id}` - получение информации о конкретном курсе по идентификатору.

Для проверки корректности HTTP-логики были написаны тесты в файле `internal/httpserver/handler_test.go`. В тестах проверяется главная страница API, healthcheck, получение списка курсов, получение конкретного курса, обработка отсутствующего курса и запрет неподдерживаемых HTTP-методов. Тестирование выполняется командой `go test ./...`, которая также используется в CI/CD-конвейере.

Следующим этапом была выполнена контейнеризация приложения. Для этого был создан `Dockerfile` с multi-stage сборкой. На первом этапе используется образ `golang:1.22-alpine`, в котором исходный код компилируется в статический бинарный файл. На втором этапе используется минимальный образ `alpine:3.20`, куда копируется только готовый бинарный файл. Также в контейнере создаётся отдельный пользователь `app`, от имени которого запускается приложение. Это делает контейнер более безопасным, так как приложение не работает от имени root-пользователя.

Для уменьшения размера Docker context был добавлен файл `.dockerignore`. В него вынесены файлы и каталоги, которые не должны попадать в контекст сборки Docker-образа, например `.git`, `.github`, временные файлы и локально собранный бинарный файл. Это ускоряет сборку и предотвращает попадание лишних данных в Docker build context.

После подготовки приложения и Docker-образа были созданы Kubernetes-манифесты. Для организации манифестов была использована структура Kustomize. Базовые манифесты расположены в каталоге `k8s/base`, а локальная конфигурация для демонстрации находится в каталоге `k8s/local`.

В базовой конфигурации `k8s/base` были описаны следующие ресурсы Kubernetes. `Namespace` создаёт отдельное пространство имён `coursevt`, в котором размещаются ресурсы приложения. `Deployment` описывает запуск приложения в Kubernetes, количество реплик, используемый контейнерный образ, переменные окружения, порты, resource requests/limits, readiness probe и liveness probe. `Service` предоставляет стабильную сетевую точку доступа к pod'ам приложения внутри кластера. `Ingress` описывает правило маршрутизации HTTP-трафика к сервису приложения по доменному имени.

Отдельно была создана локальная конфигурация `k8s/local`. Она переиспользует базовые манифесты и переопределяет параметры, необходимые для запуска в локальном Docker Desktop Kubernetes. В частности, локальная конфигурация использует Docker-образ `coursevt:local`, политику загрузки образа `imagePullPolicy: Never` и доменное имя `coursevt.local`. Это позволяет не загружать образ в внешний container registry при локальной демонстрации, а использовать образ, собранный на компьютере.

Далее был настроен CI/CD-конвейер с использованием GitHub Actions. Workflow находится в файле `.github/workflows/cicd.yml`. Он запускается при публикации git-тега, соответствующего шаблону `v*`. Это соответствует требованию о том, что сборка приложения должна запускаться при появлении git-тега. Также workflow поддерживает ручной запуск через `workflow_dispatch`.

CI/CD-конвейер состоит из нескольких этапов. На этапе `Test` выполняется checkout репозитория, установка Go и запуск тестов командой `go test ./...`. На этапе `Build and push image` выполняется сборка Docker-образа с помощью Docker Buildx, вход в GitHub Container Registry и публикация образа с тегом, соответствующим git-тегу. Например, при публикации тега `v1.0.1` собирается и публикуется образ с таким же тегом. Также дополнительно публикуется тег `latest`.

В workflow также присутствует этап `Deploy to local Kubernetes`, предназначенный для доставки приложения в локальный Docker Desktop Kubernetes. Для этого был настроен self-hosted GitHub runner. В отличие от стандартного GitHub-hosted runner, self-hosted runner выполняется на локальной машине, поэтому имеет сетевой доступ к локальному Kubernetes-кластеру. На этапе деплоя workflow собирает локальный образ `coursevt:local`, загружает его внутрь Kubernetes-ноды `desktop-control-plane`, применяет манифесты `k8s/local`, обновляет переменную `APP_VERSION` и ожидает успешного rollout Deployment.

Для локальной проверки был включён Kubernetes в Docker Desktop. После этого с помощью команды `kubectl config get-contexts` были просмотрены доступные Kubernetes context'ы, а командой `kubectl config use-context docker-desktop` был выбран локальный кластер Docker Desktop. Это позволило выполнять все последующие команды `kubectl` именно в нужном Kubernetes-кластере.

Перед развертыванием приложения был собран локальный Docker-образ командой `docker build -t coursevt:local .`. Так как Docker Desktop Kubernetes использует собственное окружение выполнения контейнеров, образ был дополнительно загружен внутрь Kubernetes-ноды. После этого локальные Kubernetes-манифесты были применены командой `kubectl apply -k k8s/local`. В результате были созданы namespace, service, deployment и ingress для приложения.

После развертывания была выполнена проверка pod'ов командой `kubectl -n coursevt get pods`. Когда pod'ы перешли в состояние `Running`, приложение было проверено через Kubernetes Service с помощью port-forward. Для этого использовалась команда `kubectl -n coursevt port-forward service/coursevt 8080:80`, после чего HTTP-запросы отправлялись на адрес `localhost:8080`.

Для проверки требования о доступности приложения по собственному доменному имени был использован Kubernetes Ingress и локальное доменное имя `coursevt.local`. После настройки локального доменного имени были выполнены HTTP GET-запросы к `http://coursevt.local/` и `http://coursevt.local/api/v1/courses`. Полученные JSON-ответы подтвердили, что приложение доступно не только через port-forward, но и через доменное имя.

Таким образом, в ходе работы были выполнены все основные этапы: разработка Go HTTP API, написание тестов, контейнеризация приложения, подготовка Kubernetes-манифестов, настройка CI/CD в GitHub Actions, запуск сборки по git-тегу и локальная доставка приложения в Kubernetes с последующей проверкой HTTP-доступности.

### Описание скриншотов

**Скриншот 1. Выбор Kubernetes context**

На первом скриншоте показана проверка доступных Kubernetes context'ов с помощью команды:

```bash
kubectl config get-contexts
```

После этого был выбран context Docker Desktop Kubernetes командой:

```bash
kubectl config use-context docker-desktop
```

Данный шаг подтверждает, что дальнейшие команды `kubectl` будут выполняться именно для локального Kubernetes-кластера Docker Desktop.

**Скриншот 2. Проверка pod'ов приложения**

На втором скриншоте показан результат команды:

```bash
kubectl -n coursevt get pods
```

Команда выводит список pod'ов в namespace `coursevt`. Успешное состояние pod'ов подтверждает, что приложение было развернуто в Kubernetes и контейнеры приложения запущены. Для корректной работы pod'ы должны находиться в состоянии `Running`, а значение в колонке `READY` должно быть `1/1`.

**Скриншот 3. Проверка приложения через port-forward и HTTP GET-запросы**

На третьем скриншоте показана проверка приложения через механизм port-forward. Для проброса локального порта на Kubernetes Service использовалась команда:

```bash
kubectl -n coursevt port-forward service/coursevt 8080:80
```

После этого были выполнены HTTP GET-запросы к приложению:

```bash
curl http://localhost:8080/
curl http://localhost:8080/healthz
curl http://localhost:8080/api/v1/courses
curl http://localhost:8080/api/v1/courses/go-basics
```

Полученные JSON-ответы подтверждают, что приложение успешно обрабатывает HTTP GET-запросы внутри Kubernetes-кластера.

**Скриншот 4. Проверка приложения по доменному имени**

На четвёртом скриншоте показана проверка доступности приложения по собственному локальному доменному имени:

```text
coursevt.local
```

Для проверки использовались HTTP-запросы через `curl`, например:

```bash
curl http://coursevt.local/
curl http://coursevt.local/api/v1/courses
```

Данный шаг подтверждает выполнение требования о доступности приложения для HTTP-запросов по собственному доменному имени. В локальной среде доменное имя `coursevt.local` используется совместно с Kubernetes Ingress.

**Скриншот 5. Выполнение CI/CD workflow в GitHub Actions**

На пятом скриншоте показан успешный запуск workflow в GitHub Actions. Workflow был запущен после публикации git-тега. В результате были выполнены этапы тестирования, сборки Docker-образа и автоматического деплоя в локальный Kubernetes через self-hosted runner.

На этапе деплоя workflow применяет локальные Kubernetes-манифесты командой:

```bash
kubectl apply -k k8s/local
```

Таким образом, в рамках работы были проверены обе части процесса: автоматическая CI-сборка в GitHub Actions и автоматическая доставка приложения в локальный Kubernetes-кластер через self-hosted runner.

## ВЫВОД

В ходе выполнения курсового проекта был разработан и проверен CI/CD-конвейер для HTTP-приложения на языке Go. В результате было создано приложение `coursevt`, которое обрабатывает HTTP GET-запросы и возвращает ответы в формате JSON. Для демонстрации работы приложения были реализованы endpoint'ы для получения информации о сервисе, проверки работоспособности, получения списка курсов и получения курса по идентификатору.

Код приложения был структурирован по архитектурным пакетам, что позволило отделить точку входа, конфигурацию, предметную область и HTTP-слой. Такой подход делает проект более понятным, расширяемым и близким к структуре реальных backend-приложений.

Для приложения был подготовлен Dockerfile, с помощью которого выполняется сборка контейнерного образа. Также были созданы Kubernetes-манифесты для развертывания приложения в кластере. В Kubernetes были использованы ресурсы `Namespace`, `Deployment`, `Service` и `Ingress`, что позволило запустить приложение, организовать доступ к нему внутри кластера и проверить HTTP-доступ через доменное имя.

CI/CD-конвейер был реализован с использованием GitHub Actions. Workflow запускается при публикации git-тега формата `v*`, выполняет тестирование приложения, сборку Docker-образа и публикацию образа в GitHub Container Registry. Это подтверждает выполнение требования о запуске сборки при появлении git-тега.

Доставка приложения в Kubernetes была реализована для локального Docker Desktop Kubernetes с использованием self-hosted GitHub runner. Такой runner выполняется на локальной машине, поэтому имеет доступ к Docker Desktop Kubernetes и может выполнять команды `kubectl`. После успешной сборки по git-тегу workflow автоматически запускает этап деплоя, применяет манифесты `k8s/local` и ожидает успешного обновления Deployment. Для внешнего Kubernetes-кластера в workflow также предусмотрен отдельный ручной deploy через `workflow_dispatch` с использованием kubeconfig.

После развертывания была проверена работоспособность pod'ов, service и ingress. Приложение успешно обработало HTTP GET-запросы через `localhost` с использованием port-forward, а также по локальному доменному имени `coursevt.local`. Таким образом, было подтверждено, что приложение доступно для HTTP-запросов после развертывания в Kubernetes.

В результате выполнения работы были выполнены основные требования курсового проекта: приложение написано на Go, обрабатывает HTTP GET-запросы, код размещён в git-репозитории, сборка запускается при появлении git-тега, подготовлена доставка приложения в Kubernetes, а работоспособность приложения после развертывания проверена через HTTP-запросы и доменное имя.
